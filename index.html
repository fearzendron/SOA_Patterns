Canonical Protocol

Dado que serviços podem aplicar diferentes tecnologias de comunicação e assim aumentar a complexidade de integração entre sistemas diferentes, o padrão Canonical Protocol sugere a aplicação de uma unica tecnologia para a comunicação.

Exemplo: SOAP sobre HTTP. Ou RESTful.

  • Wikipedia

Canonical Schema

Diferentes serviços com modelos semelhantes acabam impondo um esforço de transformação nos dados. Para resolver este ponto este padrão sugere o uso de schemas entre serviços chamados de contratos.

Exemplo: Arquivos XML como o XSD para modelos de dados e WSDL para contratos.

  • soapatterns.org
  • wikipedia

Domain Inventory

Um conjunto grande de serviços organizados sem determinar seus domínios pode trazer alta complexidade e muito esforço de todos os envolvidos em manter e evoluir. Pessoas com papéis diferentes podem acabar demandando constante alinhamento com o time por causa disso. Para resolver este problema este pattern sugere a separação dos serviços através do uso de domínios para identificar agrupamentos.

Exemplo: Na pratica podemos organizar repositorios de serviço por dominio, por exemplo, contabilidade, RH, PCP, vendas e etc.. Cada um gerenciado de forma independente.

  • soapatterns.org

Enterprise Inventory

Organizar o inventório de serviços em locais diferentes pode trazer desorganização além de possibilitar a criação de serviços iguais em repositórios diferentes por que não é possivel visualizar o todo. Este pattern visa a organização de todos os serviços em um repositório enterprize assim todos os envolvidos na organização tem um ponto central de decisão, lembrando que seguindo o pattern "Domain Inventory" os serviços em um ponto central são organizados através de domínios.

Exemplo: Softwares de governança podem realizar a organização dos serviços de forma objetiva.

  • soapatterns.org

Logic Centralization

Times diferentes criando serviços podem ocasionalmente gerar funcionalidades iguais, por exemplo, um serviço de compra de produto pode verificar o estoque, outro serviço de compras também pode verificar o estoque, se ambos os serviços implementarem a consulta ao estoque teremos redundancia. Este pattern sugere a centralização de lógicas, o controle de estoque pode virar um serviço assim pode ser reusado por outros.

Exemplo: Este item também é conhecido como Entity Services, uma entidade de domínio pode ter suas funcionalidades expostas assim sendo, ela fica centralizada e disponivel para outros serviços.

  • soapatterns.org
  • wikipedia

Service Layers

Multiplos serviços desenvolvidos por times diferentes podem gerar inconsistencias e funcionalidades redundantes, este pattern visa separar em camadas logicas determinando objetivos para cada uma delas. Usando o padrão "Logic Centralization" para determinar os objetivos de serviços para entidades de dominios, criamos uma camada Task Service que pode fazer uso de uma ou mais entity services.

Exemplo: SERASA pode expor um serviço de "consulta" que recebe um CPF, internamente este serviço chama outros serviços "composição" que juntos vão elaborar a resposta. Não é necessário neste caso expor os outros serviços.

  • soapatterns.org
  • wikipedia

Service Normalization

Serviços em um mesmo repositorio devem ser normalizados, isto significa, evitar serviços redundantes e todos os serviços precisam ter objetivos claros.

Exemplo: Dentro do domínio RH temos que ter claro quais serão os serviços, "contratar", "fecharSalarioMês" e assim por diante.
